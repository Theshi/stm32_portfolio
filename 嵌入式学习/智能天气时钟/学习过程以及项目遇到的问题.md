## 二、 核心操作数据量总结（重点物理限制）

在设计算法或编写 SPI 通信时，下面这三个操作的数据量限制是雷打不动的硬铁律：

### 1. 每次“读取”的数据量：无限制（1 ~ 8MB 均可）

- **规则**：只要你拉低片选引脚（CS=0），发送了读取指令（`0x03`）和 24 位地址，接下来只要你的单片机提供时钟（CLK），Flash 内部的地址指针就会**疯狂自动自增**。
    
- **吞吐量**：你可以一口气从 `0x000000` 连续读到 `0x7FFFFF`，把整颗芯片读穿。不需要考虑页、扇区、块的边界。
    

### 2. 一次“写入”的数据量：1 ~ 256 字节（绝不能跨页）

- **规则**：写入的核心指令叫 **Page Program（页编程，`0x02`）**。硬件限制单次写命令最多只能吃下 **256 字节**。
    
- **致命边界**：写入的地址**绝对不能跨越当前的页边界**。
    
    - _正例_：从页开头 `0x00` 写 256 字节，完美成功。
        
    - _反例_：从页尾巴 `0xF0`（第240个字节）开始写 30 字节。由于 $240 + 30 = 270 > 256$，它不会自动流向下一页，而是会触发**页内回卷**，多出来的 14 字节会跑回这页的开头 `0x00` 覆盖掉原本的数据。
        
- **算法对策**：这就是为什么我们必须用之前提到的“切面包算法（空间计算）”，把任意长度的数据切成小于等于 256 字节的小片，再一页页喂给它。
    

### 3. 每次“删除（擦除）”的数据量：只能按固定块/扇区大小

- **底层原理**：Flash 晶体管的物理特性决定了，写入只能把电平从 `1` 变成 `0`。要把 `0` 变回 `1`，必须施加高压进行“物理超度”，这个过程就叫擦除。擦除后，该区域的所有字节都会变成 **`0xFF`**。
    
- **擦除规格**（你不能指定任意大小，只能选择以下三种硬件硬编码的套餐）：
    
    1. **扇区擦除（`0x20`）**：每次必须擦除 **4 KB**。这是最精细的擦除，最适合用来做文件系统的局部更新。
        
    2. **块擦除（`0xD8` 或 `0x52`）**：每次擦除 **64 KB** 或 **32 KB**。速度比按扇区一个个擦要快得多，适合烧录大字库前清场。
        
    3. **全片擦除（`0xC7`）**：一瞬间把整颗芯片 **8 MB** 全部擦成 `0xFF`（格式化）。

## 🧠 核心算法精讲 — 两个算法让你彻底搞懂

### 🍞 算法一：不定量数据写入 — "切面包分装"

**问题：** W25Q64的页编程命令一次最多写256字节，且不能自动跨页。如果我要写1000字节怎么办？

**比喻：** 有一条长面包（你的数据），每条袋子只能装256片（一页），每16个袋子放在一个纸箱里（一个扇区=4KB）。进入新纸箱前必须把旧东西清空（擦除）。

**数学核心：**

```javascript
page_offset   = cur_addr % 256      // 当前在页内的偏移
space_in_page = 256 - page_offset   // 这一页还能写多少
chunk         = min(remain, space_in_page)  // 本轮切多少
```

**示例：从地址 0x0000F0 写入1000字节**

```javascript
第1轮: cur_addr=0x00F0, 页偏移=240, 页剩余=16  → 写16字节, 剩余984
第2轮: cur_addr=0x0100, 页偏移=0,   页剩余=256 → 写256字节, 剩余728
第3轮: cur_addr=0x0200, 页偏移=0,   页剩余=256 → 写256字节, 剩余472
第4轮: cur_addr=0x0300, 页偏移=0,   页剩余=256 → 写256字节, 剩余216
第5轮: cur_addr=0x0400, 进入新扇区→擦除扇区→写216字节, 剩余0 → 结束！
```

**1000字节被自动切成5片，跨了2个扇区，整个过程全自动！**

### 📐 算法二：空间计算 — "紧挨着码放"

**问题：** Flash 上没有"文件系统"，怎么知道新文件该写到哪个地址？

**比喻：** 货架上已有几个箱子，新箱子紧挨着最后一个箱子放。

```javascript
货架起点                                        货架终点
┌───────┬──────────┬──────┬────────────────┬──────────────┐
│ 箱子A  │  箱子B   │箱子C  │     箱子D      │   空位...     │
│ 4KB   │  12KB    │ 8KB  │     6KB        │              │
└───────┴──────────┴──────┴────────────────┴──────────────┘
0       4096      16384   24576           30720           ↓
                                                    新箱子放这里
```

**算法：** 遍历所有"已用"目录项，计算 `max(每个文件.start_addr + 每个文件.size)`，这个最大值就是下一个空闲地址。

```javascript
next_addr = max(file_end)  对所有 used==0x5A 的目录项
```

## 📖 通俗讲解 — 独立写一遍

### 第1步：理解 Flash 的物理结构

```javascript
W25Q64 (8MB)
├── Block  ×128  (每块64KB)   ← 大擦除单元
│   └── Sector ×16  (每扇区4KB)  ← 常用擦除单元（最小擦除单位！）
│       └── Page ×16  (每页256B)  ← 写入单元（最多一次写256B！）
```

**铁律：写之前必须擦除，擦除最小4KB，写最大256B且不能跨页。**

### 第2步：理解目录区设计

```javascript
Flash开头16KB = 登记本 = 128条记录 × 每条16字节
每条记录: used(1B) | id(2B) | size(4B) | start_addr(4B) | crc16(2B) | reserved(3B)
标志位: 0x00=空, 0x5A=有数据, 0xA5=已删除
```

为防止干扰误判，用魔术数字0x5A/0xA5而非0x00/0x01。

### 第3步：串起来 — 写入一个文件的完整流程

```javascript
1. 调用 W25Q64_FileWrite(id=3, data, size=5000)
2. 查目录里有没有 id=3 → 没找到
3. 找空闲目录行 → 第5行空着
4. 用"紧挨码放"算地址 → next_addr = 0x005000
5. 看空间够不够 → 5000字节，够
6. 用"切面包"写数据 → 自动切20页，跨2个扇区
7. 算CRC16 → 0x8B3A
8. 填目录（第5行）：used=0x5A, id=3, size=5000, addr=0x005000, crc=0x8B3A
9. 目录回写Flash → 完成！
```

### 第4步：自己动手写一遍的要点

1. **先写 `W25Q64_PageWrite`** — 一页256B的写入
2. **再写 `W25Q64_SectorErase`** — 4KB扇区擦除
3. **然后写切面包循环** — `while(remain>0)` 里算 `chunk`、判断擦不擦、调用 `PageWrite`
4. **最后写目录管理** — 扫描找最大尾部地址 = 下一个可写位置
5. **CRC16直接抄** — 这是一个标准算法，不需要背
6. **每个函数都先 CS↓ → 发命令 → 发地址 → 收/发数据 → CS↑**

## 🔑 关键代码片段（供你练习时参考）

**切面包循环（算法一的核心）：**

```c
while (remain > 0) {
    uint32_t page_offset  = cur_addr % 256;        // 页内偏移
    uint32_t space_in_page = 256 - page_offset;     // 页剩余空间
    uint32_t chunk = (remain < space_in_page) ? remain : space_in_page;  // 本轮写多少

    if (cur_addr / 4096 != last_erased_sector) {   // 进新扇区了？
        SectorErase(cur_addr);                      // 擦除！
        last_erased_sector = cur_addr / 4096;
    }

    PageWrite(cur_addr, data + offset, chunk);      // 写这一片

    cur_addr += chunk;
    offset   += chunk;
    remain   -= chunk;
}
```

**紧挨码放（算法二的核心）：**

```c
uint32_t next_addr = DATA_START_ADDR;   // 从数据区开头开始
for (i = 0; i < 128; i++) {
    if (目录[i].used == 0x5A) {
        uint32_t end = 目录[i].start_addr + 目录[i].size;  // 这个文件的尾部
        if (end > next_addr) next_addr = end;               // 更新最大尾部
    }
}
return next_addr;  // 这就是下一个可写地址！
```
# MCU以及FLASH烧录知识
## 一、 到底啥算“烧录程序”？

用大白话讲：**“烧录”就是给单片机装系统/拷贝软件。**

你在 Keil 里写好 C 语言代码，点击编译，电脑会生成一个 `.hex` 或 `.bin` 文件。这个文件本质上就是一堆由 `0` 和 `1` 组成的机器码。

**烧录，就是利用某种物理通道，把这堆 `0` 和 `1` 强行灌进单片机内部的 permanent 永久存储区（也就是 Flash 闪存）里。**

一旦烧录成功，哪怕你断电、拔线，程序也跑不掉。下次一上电，单片机的 CPU 就会雷打不动地去 Flash 的起点读取这些指令开始运行。你在 Keil 里点“Download（下载）”，本质上就是进行了一次烧录。

---

## 二、 烧录程序的三大主流方式

单片机是怎么把数据吃进去的？主要有三种通路。这时候就能引出 **FlyMcu** 了。

### 1. ICP 方式（在电路编程）—— 研发调试最常用

- **通道**：通过专用的**仿真器/下载器**（如 ST-Link、DAP-Link、J-Link）。
    
- **特点**：直接用物理触手死死抓住单片机内核的调试管脚。你在 Keil 里写完代码，啪一下点那个“魔法棒”旁边的 **Download** 按钮，程序就顺着仿真器直接打进 Flash 里的方式，就叫 ICP。
    

### 2. ISP 方式（在系统编程）—— FlyMcu 的舞台

- **通道**：不需要仿真器，只需要最普通的**串口（USB转TTL，比如 CH340 芯片）**。
    
- **什么是 FlyMcu**：它就是电脑端的一款免费的**串口烧录软件**。
    
- **工作原理**：单片机出厂时，官方在芯片内部一块不能修改的区域（ROM）里预装了一小段官方代码（叫 Bootloader）。当你把单片机的 `BOOT0` 引脚拉高、`BOOT1` 引脚拉低，然后复电，单片机就会启动这段官方代码，开始监听串口1（PA9/PA10）。
    
    这时候你打开 **FlyMcu** 软件，选中你的 `.hex` 文件，点击开始，FlyMcu 就会顺着串口把数据发过去，单片机自己内部接应并写入 Flash。
    

### 3. IAP 方式（在应用编程）—— 商业产品远程升级

- **通道**：利用你写好的上层应用程序升级（比如用蓝牙、Wi-Fi、或者我们之前聊的 YMODEM 串口协议）。
    
- **特点**：单片机自己运行着程序，自己去网上下载新固件，然后自己把自己擦除并写入新程序（自己升级自己）。
    

---

## 三、 傻傻分不清：JTAG、SWD、DAP、ST-Link 到底啥区别？

我们在 Keil 调试时，经常看到这些缩写混在一起。其实它们根本不是一类东西，它们的关系是：**“通信协议”与“硬件外壳”的关系。**

我们可以把它们分成两组：

### 第一组：通信协议 / 接口标准（相当于：马路上的交规）

这是单片机和外界对话的**语言通路标准**。

- **JTAG**：老一代的标准协议。功能极度强大，不仅能下载程序、在线仿真，还能测试芯片管脚的物理状态。**缺点是太浪费引脚**，需要用到 4~5 根线。
    
- **SWD (Serial Wire Debug)**：ARM 公司专门为微控制器定制的**现代级大杀器协议**。它只需要 **2 根线**（SWDIO 数据线、SWCLK 时钟线）。由于极度省引脚，现在 STM32 开发中 **99% 都在用 SWD**。
    

### 第二组：仿真器硬件（相当于：跑在马路上的卡车）

这是你握在手里、插在电脑 USB 口上的那个**物理硬件小方块**。它们内部的代码实现了上面的协议。

- **ST-Link**：ST（意法半导体）官方的亲儿子。只能用来烧录和调试 ST 自家的芯片（如 STM32）。它同时支持 JTAG 和 SWD 协议。
    
- **J-Link**：德国 SEGGER 公司做的“行业老大哥”。性能极其彪悍，支持全球几乎所有架构的单片机，速度飞快。缺点是一个正版要几千块（市场上充斥着盗版）。它也同时支持 JTAG 和 SWD。
    
- **DAP-Link (CMSIS-DAP)**：ARM 官方开源的**功德无量级下载器**。
    
    - **特点**：它是开源的，谁都可以做，所以硬件只要十几块钱。
        
    - **最大优势**：**免驱动！** 它走的是电脑的 USB-HID 协议，插上电脑就能直接用，绝对不会像 ST-Link 那样经常遭遇“驱动冲突”或者“固件需要升级”的折磨。它同样完美支持 SWD 协议，在 Keil 里原生完美兼容。
